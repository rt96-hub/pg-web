# 019 — Roadmap resequencing: pull the backup/export story forward, plus ecosystem & launch motions

**Status:** Open strategic memo — for maintainer review, then convert sub-items into scheduled work
**Date opened:** 2026-06-11
**Author:** Handoff prompt (derived from external codebase analysis, 2026-06-11)
**Prerequisites:** none — this frames sequencing for 013–018 and 020
**Context:** pg-web went public this week (crates.io `pg-web` v0.2.0, Docker Hub `rtaylor96/pg-web`, and the dogfooded docs site at pg-web.dev) with Phase 1 complete and Phase 2 in planning. The roadmap is feature-honest and well-ordered *within* each phase, but several high-leverage items are filed by topic rather than by marketing value or by dependency — most notably a "hand someone one file and they have the whole running app" backup story buried in Phase 4. This memo recommends a resequencing, names the gaps that aren't on the roadmap at all, and proposes the ecosystem/launch motions a freshly-public, bus-factor-1 project needs. It recommends sequencing only; it proposes nothing that violates a stated non-goal.

---

## Summary

The roadmap (`docs/ROADMAP.md`) is disciplined and credible. The phase order — Synchronous Core (done) → Auth/RLS/realtime → Async jobs → Observability — is a reasonable narrative of capability. But "narrative of capability" is not the same as "order that maximizes adoption and de-risks the next phase," and on three counts the current order leaves value on the table:

1. **The single most marketable, least-contested idea on the whole roadmap — "the app you can `pg_dump`" — is filed under Phase 4** (`ROADMAP.md:83-85`, `255-257`), behind auth, realtime, schema-diffing, and the async job queue. It is also the *cheapest* of the three to ship a useful slice of. **Lean:** promote operational `backup`/`restore` and `export --code-only` to near-term and make them a headline story.

2. **Phase 2 (auth/RLS) has two unacknowledged hard prerequisites sitting under it** — the response contract (prompt 013) and an execution-role/privilege floor (prompt 014). Without 013 there is no `Set-Cookie`, so cookie sessions cannot exist; without 014 handlers run as superuser, so RLS does not actually enforce. The roadmap treats Phase 2 as the next move (`README.md:83`, session_6) without surfacing that it ships compromised if these don't land first. **Lean:** 013 and 014 are Phase-2-gating, not optional polish — schedule them as the front of Phase 2.

3. **Several genuinely-needed pieces are on no phase at all** — a published benchmark to back the VISION throughput claim, request-path caching, the full HTTP method set + uploads, an umbrella security-hardening pass, extension upgrade scripts, and `pg-web check` accuracy. These map to prompts 015–018 and should be tracked rather than discovered later.

Plus: a freshly-public solo project with **2 stars and 0 open issues reads as "finished / abandoned," not "healthy."** The cheapest adoption lever available right now is community legibility — seeded issues, a second flagship example, an honest comparison page, and non-Docker packaging — and none of it competes with feature work for the same risk budget.

This memo does not introduce a single capability that violates the project's non-goals (`ROADMAP.md:87-95`, `295-302`; `VISION.md:46-52`): no ORM, no managed-DB support, no GraphQL, no non-HTMX frontend, Postgres-only. Everything here is sequencing, packaging, positioning, and dogfooding of features the project already intends to build.

## The headline opportunity: "the app you can `pg_dump`"

The roadmap already contains the best growth story pg-web has, and it is undersold by its placement. Three related items live in Phase 4 (`ROADMAP.md:83-85`, expanded at `255-257`):

- **`pg-web backup` / `pg-web restore`** — thin wrappers over `pg_dump --format=custom` / `pg_restore`, target resolved from `pgweb.toml`. Because routes, templates, assets, handlers, schema, *and* user data all live in one database, one dump captures the entire application, and `restore` reboots it into a working app.
- **`pg-web export --code-only` / `pg-web import`** — a portable, data-free dump of just the app's *shape*: `pgweb.routes` + `pgweb.templates` + `pgweb.assets` + the `pgweb.pages__*` handlers + `pgweb.settings` (minus secrets). For cloning a prod app's code into a dev DB, or a git-tag-stable snapshot of the live app that survives a DB rebuild.
- **`pgweb.sources` schema** — mirror the working tree (and optionally `.git/` history) into framework tables so a single `pg_dump` is a *runnable-from-dump* snapshot: schema + data + app code + version history in one file.

Here is why this is the headline, not a Phase-4 footnote.

**It is the project's only truly category-defining claim.** Everything else pg-web does has a credible competitor (SQLPage renders SQL→HTML; PostgREST exposes tables as REST; Omnigres also runs HTTP in Postgres). But "**hand someone a single file and you've handed them the entire product — code *and* state — restorable into a fresh container as a running system**" is a sentence no mainstream web framework can say. Rails/Next/Django all scatter the app across a code repo, a separate database, an asset pipeline, and environment config; reconstituting a running system means orchestrating all four. pg-web's "everything in one database" thesis makes "clone prod to a laptop in one command" and "the app *is* the dump file" literally true. That is a demo that sells itself — it is visceral in a way that "zero network hops" (true, but abstract) is not.

**It is the least-contested idea on the roadmap.** It violates no non-goal, touches no invariant, needs no new request-path code, and has essentially no design risk for the smallest slice — `backup`/`restore` is "shell out to `pg_dump`, resolve the target you already resolve for `push --target`." Compare that to auth (cookie format debates, secret rotation, RLS correctness) or schema-diffing (the heaviest single feature below Phase 2, `ROADMAP.md:225`). The marketing payoff is inversely correlated with the engineering risk here, which is exactly the profile you want to pull forward.

**It is the cheapest of its three pieces that matters most.** The roadmap itself already splits these by cost (`ROADMAP.md:263`): operational `backup` and `code-only export` are "the smaller, ship-sooner pieces"; `pgweb.sources` is "the full version." The mistake is filing all three at the same depth. The two small ones could ship in a focused session; the big one can stay a later phase.

**Lean:** Promote **`pg-web backup` / `restore` and `export --code-only` / `import` to near-term** — runnable post-Phase-2, or even in parallel with it since they touch the CLI and `pg_dump`, not the request path or the auth plumbing. Make "the web framework you can `pg_dump`" a headline marketing line (see Positioning). Keep **`pgweb.sources`** in Phase 4 (or its own phase) until the open questions below are answered — it is the part with real cost and real risk.

**Dependencies, risks, and the things that make this non-trivial despite the easy framing:**

- **Storage cost of `pgweb.sources`.** A busy `.git/` is gigabytes; mirroring full git objects into a table bloats every dump and every restore. The roadmap already flags this (`ROADMAP.md:257`). **Lean for the eventual `sources` work:** default to **working-tree-only** (`files(path, content, mode, mtime)`), make git-history mirroring strictly opt-in, and never let it touch the small `backup`/`export` commands that ship first.
- **Keeping `pgweb.sources` in sync without slowing the dev loop.** Pushing file blobs on every `pg-web push` would tax the hot iteration path the whole framework is built around (Blake3-deduped, debounced, transactional — `ROADMAP.md:326`). **Lean:** decouple it — a separate `pg-web sync-source` (manual or git-`post-commit`-hook-driven), *not* an extension of the per-save `push`. This is an open question the `sources` design must answer before scoping (`ROADMAP.md:257`).
- **Secrets must be excluded from code-only exports by default.** `export --code-only` dumps `pgweb.settings`, which is also where user-set secrets live (`pg-web env set`, `ROADMAP.md:34-35`; framework secret in `session_6.md:46`). The roadmap already says "minus secrets" / "minus reserved-key `env`" (`ROADMAP.md:84`, `256`) but this must be the **safe default**, with `--include-secrets` an explicit opt-in, not the reverse. This cross-references **prompt 014** (privilege / secrets handling) directly: whatever 014 decides about how secrets are stored and scoped governs what `export` is allowed to emit. Get the default wrong and the headline feature becomes a credential-leak vector — a "hand someone the file" story is exactly the wrong place to leak `framework_secret` or a Stripe key.
- **`restore` is not free of footguns either.** Document loudly that restoring a custom-format dump into a fresh container reboots the app *including its `shared_preload_libraries` assumptions and its `framework_secret`* — i.e. a restored prod dump carries prod's session-signing secret, so "clone prod to laptop" also clones the ability to mint prod session cookies. That is fine for a true backup, dangerous for sharing. The `backup` (full, secrets included) vs `export --code-only` (shape, secrets excluded) split the roadmap already drew (`ROADMAP.md:256`, "Distinct from operational backup") is exactly the right mental model — the docs just need to make the security difference between them unmissable.

**Three sequencing arguments worth stating plainly, since they are the crux:**

- **013 + 014 are Phase 2's foundation, and the roadmap doesn't say so.** session_6 specs a login handler that sets a cookie (`session_6.md:49`) and a CSRF token set on every response (`session_6.md:111`) — both impossible today (no header channel; 013 exists precisely to add one). And the RLS Bridge (`ROADMAP.md:186-190`) is only a *real* security boundary if handlers do **not** run with RLS-bypass privileges — which is 014's entire subject. **If Phase 2 starts before 013/014, it ships an auth story whose cookies can't be set and whose row-level isolation doesn't isolate.** The phase-discipline culture (`CLAUDE.md:51`) is the right instinct here: these are the staging work that lets Phase 2 be done *correctly* rather than *first*.
- **The async job queue is mis-filed as "scale" when it is really "unlock."** Its phase-3 placement reads as a performance concern, but the actual gated features are functional: external API calls (`ROADMAP.md:170`), OAuth, email verification, password reset (`session_6.md:58-59`, `168`), and webhooks. Several of these are the *obvious* next things a user will want after auth. Worth re-labelling so its priority is legible — it may deserve to move up if those deferred features start drawing demand.
- **`/_pgweb/health` should not be hostage to the dashboard.** It is currently only mentioned alongside the Phase 3 metrics endpoint (`ROADMAP.md:239`) and the Phase 4 admin UI (`248`). A liveness probe is a deploy-day necessity and a few lines of code; ship it the moment anyone runs pg-web behind a real load balancer.

## Gaps not on the roadmap at all

These are real, scoped-in-prompts (or implied by session_6) but absent from any phase in `docs/ROADMAP.md`. Brief enumeration — each is a candidate GitHub issue (see Ecosystem) and most are already framed in a sibling handoff prompt:

- **Response contract v2 (013).** Status codes / headers / cookies / redirects / content-type for dynamic responses. *The* Phase-2 prerequisite. Already written up as prompt 013.
- **Privilege floor + threat model (014).** A non-superuser execution role for handlers and a written threat model. Without it, RLS (Phase 2) is decorative and "the app you can `pg_dump`" can leak secrets. Belongs to a security-hardening umbrella (below).
- **Published benchmark (015).** Make the `VISION.md:58` 1k-req/s claim measured rather than aspirational. Honesty-brand-critical.
- **Request-path caching (016).** Dynamic responses currently cannot be cached at all (no `Cache-Control` channel until 013).
- **Full HTTP method set + uploads + compression (017).** PUT/PATCH/DELETE, multipart uploads, body-size limits, gzip/br. Today's surface is effectively GET/POST + static assets.
- **Extension upgrade scripts (018-adjacent).** `CREATE EXTENSION ... UPDATE` migration path. 013 *itself* flags this as an open question (`013:249`, `188`): the moment the project ships a second extension version with new install-SQL helpers, existing installs need an upgrade path, not just install-from-scratch. There is no roadmap item for "how do extension versions upgrade in place."
- **Security-hardening umbrella (CSRF + rate-limiting + security headers + privilege floor).** session_6 plans CSRF (`session_6.md:108-121`) and *documents* (but does not ship) login rate-limiting (`session_6.md:61`, "framework-provided rate limit is Phase 4") and leaves security headers (HSTS/CSP/`X-Content-Type-Options`/frame options) unaddressed. There is no single "harden the framework" pass on the roadmap — these are scattered across phases and open questions. Recommend an explicit hardening milestone that gathers: 014 privilege floor, CSRF, framework-set security headers (via the 013 header channel), rate-limiting primitives, and the secrets-in-export default. Pairs naturally with 015's threat-model deliverable.
- **`pg-web check` accuracy.** The offline validator uses the generic `sqlparser` crate with a Postgres dialect (`ROADMAP.md:38`, `162`) and has already shipped false-positive bugs against real Postgres SQL — see existing `prompts/001` (comment/literal parsing), `prompts/003` (GIN/`gin_trgm_ops` opclass). `prompts/004` already proposes the fix: a **live-DB preflight** that validates against an actual Postgres instead of a third-party parser that will always lag Postgres's grammar. Reference 004; this is a correctness-of-tooling gap, not a feature gap, and it erodes trust in the one command that is supposed to catch mistakes before they hit the DB.

## Ecosystem & community motions

The project is **bus-factor-1 with 2 stars and 0 open issues**. For a freshly-public repo that combination does not read as "small but healthy" — it reads as "a finished thing one person made," which is a discouraging signal to a would-be contributor or adopter trying to gauge momentum and tractability. The cheapest available adoption levers are about *legibility*, and none of them compete with feature work for the same risk budget.

- **Seed GitHub issues from the roadmap.** An empty issue tracker is the single loudest "this is dead/done" signal. Convert the Gaps above and the smaller roadmap items into issues so a newcomer sees concrete, tractable, *labeled* work. Mix in genuinely-small ones (`/_pgweb/health`, a `pg-web check` false-positive repro from `prompts/001`/`003`, a docs fix) so the tracker shows both vision and footholds. This is an afternoon of work with outsized signaling value.
- **A second flagship example beyond `examples/todo`.** One example reads as "a toy that proves the tutorial"; two reads as "a framework you can build different things with." The natural choice is **blog-with-auth** — which is exactly prompt **020 (site-v2)** — because it *dogfoods Phase 2* (auth, sessions, RLS, per-user content) and satisfies the CLAUDE.md companion-app rule (`CLAUDE.md:50`) for that phase in the same stroke. One artifact, three jobs: example, dogfood, and proof that auth works in anger.
- **An honest comparison page.** Prospects will ask "how is this different from SQLPage / PostgREST / Omnigres?" — answer it before they have to guess. A fair, non-strawman page: **SQLPage** (SQL→HTML components, but a component DSL rather than your own Tera templates, and no in-Postgres HTTP worker), **PostgREST** (auto REST over tables — complementary, not competing; pg-web is HTML-first and ships logic+UI+assets in the DB), **Omnigres** (also HTTP-in-Postgres — the closest architectural neighbour; differentiate on the HTMX-first app model, the `pages/`-as-route DX, and the single-binary CLI loop). Leading every row: pg-web's "the app you can `pg_dump`" artifact story, which none of the three offer. Honesty is the brand (the docs already say "Dev UX that doesn't lie," `README.md:10`) — a comparison page that concedes where competitors are stronger will be *more* persuasive, not less.
- **Non-Docker packaging so `CREATE EXTENSION` works on a user's own Postgres.** Today the only supported runtime is the bundled image (`pgweb/postgres` / `rtaylor96/pg-web`). That excludes everyone who already runs their own Postgres 15/16/17 and just wants the extension. Publish the extension through the Postgres-native distribution channels: **PGXN**, **pgxman**, and/or **Trunk** (`pgrx` extensions package cleanly to all three). The CLI already ships on crates.io (`cargo install pg-web`); pairing it with `CREATE EXTENSION pg_web_ext` on a BYO Postgres — no bundled image required — dramatically widens the addressable install base and reinforces the "it's just a Postgres extension" pitch. (Respects the managed-DB non-goal, `ROADMAP.md:91` — this is for self-hosted Postgres, which the project already targets.)
- **A `CONTRIBUTING`-grade "good first issue" set.** `CONTRIBUTING.md` is referenced (`README.md:78`) but the on-ramp needs a curated handful of small, well-described, genuinely-isolated issues labeled `good first issue` — the `pg-web check` false-positives and the `/_pgweb/health` endpoint are ideal: real, bounded, and reviewable. This is the difference between "drive-by contributor bounces" and "drive-by contributor lands a PR."
- **Resolve the temporary Docker Hub namespace.** The README flags `rtaylor96/pg-web` as temporary three times (`README.md:13`, `15`, `97`) pending the official `pgweb` org. Every "(temporary)" in published docs is a small credibility tax and a future breaking-change for early adopters' compose files. Claim `pgweb` (or decide the permanent name) and cut over before the install base grows — the cost of migrating users rises monotonically with adoption.

## Positioning & messaging

**Lean (the one-line recommendation):** lead with the artifact, not the latency — **"the web framework you can `pg_dump`: your whole app, code and data, is one Postgres database you can hand to anyone."** Pair it with the dogfooded site (pg-web.dev is itself a pg-web app, `README.md:62`, `99`) as living proof.

The reasoning:

- **Lead with the artifact story.** "The app you can `pg_dump`" (backup/export, promoted per the headline section) is the claim no competitor can make and the demo that lands viscerally. "Zero network hops / SPI access" is *true and important* but abstract — it is a great *second* point, the explanation of *why* the architecture is interesting, not the hook that makes someone clone the repo.
- **Lead with the dogfooded site as proof-of-life.** A docs site that *is* a pg-web app is the strongest possible "this is real, not a README-ware framework" signal. Foreground it.
- **Do NOT headline raw latency or throughput until the runtime can cash the check.** `VISION.md:58` claims 1,000 req/s on a 1-vCPU VPS, explicitly tagged "(Target — to be benchmarked)." The runtime is a **single-threaded Tokio runtime** (`OVERVIEW.md:119`) with **one HTTP request = one SPI transaction** (`CLAUDE.md:39`). That may well sustain the target — but until prompt **015** measures it, quoting the number is exactly the kind of unbacked performance claim the project's honesty brand exists to avoid. Performance is a *later* headline, earned with a published benchmark.
- **Extend the honesty brand to performance and security.** The docs already differentiate on candor ("Dev UX that doesn't lie," `README.md:10`; the roadmap's explicit deferrals and "to be benchmarked" tags). The highest-integrity version of that brand publishes a **benchmark** (015) and a **threat model** (014/hardening umbrella) *before* leaning on performance or security as selling points. "Here is exactly how fast it is and exactly what it protects against" is a stronger position than silence on both — and it is consistent with everything the project already does.

## Research tasks

For the maintainer (or a future session) converting this memo into scheduled work:

1. **Validate the backup/export cost model.** Take a representative `pg-web` app, `pg_dump --format=custom` it, and measure: dump size with data, dump size code-only, and restore time into a fresh container. Confirm the "one file = running app" demo actually works end-to-end (restore a dump into a clean `pgweb/postgres` container and hit it with `curl`). This is the demo that anchors the positioning — verify it before marketing it.
2. **Audit the current handler execution privilege.** Determine empirically what role/privilege `pgweb.pages__*` handlers run as inside `BackgroundWorker::transaction` today, and write a minimal RLS test proving whether a policy keyed on `pgweb.user_id` *actually* blocks cross-tenant reads under the current execution role. This is the evidence base for prompt 014 and the gating argument for Phase 2.
3. **Survey the PGXN / pgxman / Trunk publishing flow for a `pgrx` extension.** Confirm which channels accept a `pgrx`-built extension cleanly, what metadata each needs, and whether `pg_web_ext` builds against stock PG 15/16/17 packages without the bundled image. Output: a concrete packaging plan (which channel(s), what CI changes).
4. **Benchmark harness scoping (feeds 015).** Decide the benchmark methodology — workload (the VISION "fetch and render" against `examples/todo`), hardware (1-vCPU/2-GiB to match the claim), tool (`wrk`/`oha`/`vegeta`), and what to report (p50/p95/p99, req/s, error rate under load). This is prompt 015's input; this memo just flags that it must precede performance messaging.
5. **Competitive landscape verification for the comparison page.** Confirm current capabilities of SQLPage, PostgREST, and Omnigres (especially Omnigres, the closest neighbour) so the comparison is fair and non-strawman. Note specifically whether any of them offer a "one-file runnable snapshot" story (they do not, as of analysis — verify).
6. **Confirm the extension-upgrade question (feeds 018-adjacent / 013).** Determine whether any `CREATE EXTENSION ... UPDATE` path exists today (013 open question #8, `013:249`). If not, scope the minimal version-upgrade mechanism, since 013 ships new install-SQL helpers that existing installs will need.

## Open questions

For the maintainer — these are the judgment calls this memo deliberately leaves open:

1. **How aggressively do we promote backup/export?** Full headline-marketing push around "the app you can `pg_dump`" now (with only the `backup`/`restore` + `export --code-only` slice shipped), or build the slice quietly first and market once `pgweb.sources` makes the "code + history + data in one file" story complete? Marketing the partial story risks "that's just `pg_dump`" dismissals; waiting risks leaving the best hook unused during the post-launch attention window.
2. **Do we delay Phase 2 for 013 and 014, or start Phase 2 and backfill?** This memo argues 013/014 are gating (auth can't set cookies; RLS won't enforce). Do you accept that and front-load them, or start the session_6 tracks and accept that the cookie mechanism + privilege floor land mid-phase? The risk of the latter is shipping a `v0.3.0` whose security story has a hole that's hard to walk back once apps depend on it.
3. **Docker Hub namespace timing.** Claim `pgweb` (or decide the permanent name) and cut over now while the install base is ~tiny, or wait until the org is sorted and accept that every early adopter's compose file references the temporary `rtaylor96/pg-web` and will need migrating? The migration cost only grows with adoption.
4. **How much to invest in community vs. features pre-traction?** With bus-factor-1 and limited time, what's the split between (a) seeding issues / comparison page / packaging / second example — which buy *legibility and adoption surface* — and (b) shipping Phase 2 — which buys *capability*? Is the goal right now to attract contributors/users, or to deepen the framework and attract later?
5. **Is `pgweb.sources` a Phase-4 deliverable or its own phase?** The roadmap leaves this open (`ROADMAP.md:257`). Given its unresolved storage/sync/git-packing questions, does it warrant a dedicated phase (like schema-diffing got, `ROADMAP.md:202-225`), or fold into Phase 4 observability?
6. **Does the security-hardening umbrella warrant its own milestone?** CSRF, rate-limiting, security headers, the 014 privilege floor, and the secrets-in-export default are currently scattered across Phase 2, Phase 4, and open questions. Gather them into one explicit "harden the framework" pass (paired with the 015 threat model), or keep them distributed across the phases that touch them?
7. **Non-Docker packaging — which channel(s), and is it worth the CI cost now?** PGXN, pxman, and Trunk are not free to set up and maintain. Pick one as the canonical channel, do all three, or defer until someone actually asks for BYO-Postgres install? (Research task 3 informs this, but the prioritization is yours.)
8. **Should the async job queue move up?** Given that it unblocks OAuth, email verification, password reset, and external API calls — several of which are the obvious post-auth asks — does it deserve to jump ahead of some Phase 2/observability work, or does it stay Phase 3 and let the deferred features wait?
