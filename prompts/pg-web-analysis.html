pg-web — full codebase analysis

A PostgreSQL extension that turns the database itself into the web server: an Axum HTTP listener runs inside a Postgres background worker, routes and templates live in tables, business logic is SQL, the UI is HTMX, and a Rust CLI owns the entire dev/deploy loop.

Analyzed from a full clone of rt96-hub/pg-web @ 918f40b (2026-06-10) · every source file read · live checks against crates.io, Docker Hub, and pg-web.dev

§Executive summary

pg-web is a real, working, unusually well-engineered implementation of a radical idea — and it is one week past its public launch.

The thesis ("PostgreSQL is your web server — zero hops between web tier and data") is fully realized for its Phase 1 scope: you can cargo install pg-web, scaffold an app, get a live-reloading dev loop, and deploy a working HTMX app as a single Docker container. The project's own documentation site at pg-web.dev is served by pg-web itself, in production mode, with fingerprinted assets — the strongest credibility signal a framework this young can have.

The engineering hygiene is far above what its size suggests: 230 tests across five mandatory tiers, a typed error catalog, a dated architectural decision log, honest "known limitations" sections, and a fully documented AI-agent-driven development process (this repo is as much a showcase of disciplined LLM-assisted engineering as it is a web framework).

Its ceilings are equally real: requests are processed one at a time on a single thread with no timeout; every handler runs as the Postgres superuser; and handlers cannot set status codes, headers, cookies, or content types — which quietly blocks the planned Phase 2 auth work and the advertised "JSON API" use case. None of these are fatal, all are fixable, and most are already named somewhere in the project's own roadmap. The gap between the vision's ambition and the runtime's current capabilities is the central tension of the project.

Verdict: a promising, genuinely differentiated framework in the "Postgres-as-platform" wave — today best treated as an excellent tool for small self-hosted apps, internal tools, and agent-era experiments, with a credible (but security- and concurrency-gated) path to more.

Workspace: two strictly-decoupled crates — pg_web_ext (the extension: HTTP worker, router, Tera, SSE livereload) and pg-web (the CLI: init/up/down/dev/push/migrate/env/check). They communicate only through six framework tables in the pgweb schema: routes, templates, assets, migrations, settings, deployments.

1What the project is

pg-web collapses the standard four-layer web stack (browser → load balancer → app server → ORM → database) into one process tree. When Postgres boots, the extension registers a background worker — a dedicated OS process forked by the postmaster — which runs a single-threaded Tokio runtime serving HTTP on :8080. Every request is resolved by querying framework tables over SPI (Postgres's in-process Server Programming Interface): look up the route, fetch the Tera template, call the user's SQL handler function, render, respond. No TCP between web tier and database, no connection pool, no ORM.

Architecture at a glance

The developer contract

Apps are pure SQL + HTML + a small TOML file. The filesystem layout is the router:

my-app/
  pages/
    index.html + index.sql          → GET /          (JSON → Tera pipeline)
    todos/post.sql + post.html      → POST /todos    (HTMX fragment response)
    todos/[id]/index.sql            → GET /todos/:id (req.path_params.id)
    _404.html                       → custom 404
  public/styles.css                 → BYTEA-backed asset, ETag/fingerprint cached
  migrations/0001_create_todos.sql  → forward-only raw SQL, ledgered
  pgweb.toml                        → env = development | production

Every handler is a Postgres function with one uniform signature: pgweb.pages__<name>(req json) RETURNS json|text, where req carries body, query, method, path, path_params. If an .html sibling exists, the JSON return feeds a Tera template; if not, the text return is sent as-is. Validation is delegated to database constraints, with PL/pgSQL EXCEPTION blocks turning check_violation into inline HTMX error fragments — a genuinely elegant pattern the demo app exercises.

What a request costs

Per GET: one SPI transaction wrapping (a) a full scan + specificity sort of all routes for the method, (b) a template fetch, (c) the handler call through a PL/pgSQL wrapper that catches exceptions structurally — plus a second small transaction to read pgweb.settings.env for live-reload injection, and a fresh Tera::one_off parse of the template every time. All documented as deliberate Phase 1 simplicity with named re-evaluation triggers (e.g., "revisit the route scan above 1,000 routes").

3How it's being built — the agentic development model

This repo is one of the cleaner public examples of LLM-driven engineering with real guardrails, and that's worth analyzing in its own right:

  • CLAUDE.md is a genuine north-star prompt: seven numbered architectural invariants ("one HTTP request = one SPI transaction", "extension ↔ CLI strictly decoupled", "no raw FFI"), coding practices, commit conventions, and session rituals ("if your change touches an invariant, raise a flag and wait for human confirmation").
  • prompts/001–012 are numbered work orders — each a structured spec (status, context, research tasks, constraints) for a future agent session. Several explicitly instruct the agent to research and discuss with Robert before designing anything.
  • docs/internal/sessions/session_1–6.md are session logs: what shipped, what was deferred and why, plus a numbered pitfalls catalog (#1–18) of environment landmines (pgrx ICEs, Git Bash path mangling, port shadowing) so future sessions don't rediscover them.
  • The cadence it produced: v0.1.0 — a usable framework with five test tiers — in seven calendar days across 4 sessions; v0.2.0 the next day; then a ~5-week pause; then a 2-week public-launch push.

Two honest caveats. First, single-author + agent-generated means the code has likely never had an adversarial human review — consistency is excellent, but consistent blind spots are exactly what this process produces (the superuser-execution issue in §5 is a plausible example). Second, session logs and prompts double the repo's documentation mass; the discipline of keeping OVERVIEW.md regenerated at milestones is currently holding, but doc drift is already visible in small ways (the architecture doc claims a pgweb.port GUC that doesn't exist yet).

4Strengths

1 · The whole loop actually works

Not a tech demo: cargo install pg-web → init --template todo → up → migrate apply → push → working app. Dev mode adds watch, content-hash dedupe, SQL preflight in BEGIN…ROLLBACK, and browser live-reload (CSS changes hot-swap without a reload). The deploy story (Caddy + compose + SSH-tunneled push) is documented end-to-end and demonstrably used for pg-web.dev.

2 · Push is a real deploy tool

One transaction: validate templates pre-DB, upsert routes/templates/handlers/assets, verify every handler exists with the right signature, reconcile-delete anything no longer on disk, sync settings, append a deployments-ledger row. Plus --dry-run (rollback), a migration gate (refuses to push past pending migrations), jittered retry on concurrent-DDL races, and a pg_stat_activity-powered diagnostic that names the sibling pusher and the exact kill command. This is more operational care than many funded frameworks ship.

3 · Error experience

A flat, typed error catalog (PGWEB_E001–E999) where every variant carries a stable code, title, remedy text, and a hand-rendered (deliberately Tera-free) dev page showing SQLSTATE/MESSAGE/DETAIL/HINT/CONTEXT plus the request dump — all HTML-escaped. Production mode logs the full picture and serves an opaque 500. The classification even distinguishes Tera parse errors from render errors so the fix hint differs.

4 · Testing discipline

Five tiers with explicit philosophy: pgrx tests against real Postgres (never mocked), HTTP smoke, CLI unit/integration, testcontainers Docker E2E against the actual shipped image, and a 19-section black-box CLI smoke script. E2E covers the ugly paths: broken templates rejected at push, prod-mode information hiding, ETag/304, concurrent-push retry, livereload SSE end-to-end.

5 · Architectural restraint

Axum used as a deliberately thin shell (one fallback handler) so migrating off it stays cheap. Naïve route scan chosen with a written re-evaluation trigger. No premature caching, no premature abstraction, dead code kept only with a documented Phase-2 reason. The "decision log with rejected alternatives" habit means future contributors inherit the reasoning, not just the result.

6 · Honest documentation

The docs repeatedly flag their own gaps: "push is localhost-only until F.2", "live-reload lacks bfcache simulation tests", "sqlparser is less authoritative than Postgres's own parser", "secrets are not encrypted at rest." Prompt 005 even documents a docs-vs-reality contradiction (advertised JSON APIs vs. hardcoded text/html). This level of self-awareness materially raises trust in everything else the docs claim.

5Problems & risks

Each finding is graded by impact on the project's stated ambitions (a production-deployable multi-user framework). Click to expand. File references point into the repo at 918f40b.

Architecture & performance

▶criticalThe server handles exactly one request at a time — and a slow handler stalls everything, forever

The worker builds a current_thread Tokio runtime (required: only the SPI-attached thread may touch Postgres), and every request runs BackgroundWorker::transaction(...) — a synchronous, blocking call — directly on that one event-loop thread. While any handler's SQL runs, nothing else makes progress: not other page requests, not asset serving, not livereload SSE keepalives.

Compounding it, there is no per-request timeout — no statement_timeout is set around handler calls. A handler that hits a lock, a slow plan, or a pg_sleep() wedges the entire web tier indefinitely. The docs acknowledge external-API blocking as a Phase-3 item, but the actual behavior is stricter than documented: it isn't "blocks the worker thread", it's "serializes the whole site."

The VISION target of 1,000 req/s on 1 vCPU may still be reachable for sub-millisecond SPI handlers sequentially, but tail latency under mixed traffic will be poor, and no benchmark exists yet. Paths forward exist (N background workers sharing the port via SO_REUSEPORT, each with its own SPI backend; or a bounded queue with backpressure — already sketched as Phase 3 "internal concurrency management"), but this is the framework's load-bearing constraint.

Evidence: crates/pg_web_ext/src/worker.rs:63-76 (current_thread runtime), src/router.rs:67-71 (blocking transaction per request), src/http.rs:120 (await-free dispatch); no statement_timeout anywhere in the ext crate.

▶criticalEvery handler executes as the Postgres superuser — which also quietly breaks the Phase 2 RLS plan

The worker attaches to SPI with connect_worker_to_spi(Some(&db), None) — a NULL username, which in Postgres means the background worker runs as the bootstrap superuser. Consequences:

  • Any SQL injection bug in user-written handler code (string-concatenated SQL in someone's PL/pgSQL) is instantly a full-cluster compromise — there is no privilege floor beneath app code.
  • Every handler can read pgweb.settings (where the framework tells you to put Stripe keys), drop schemas, or alter the framework's own tables.
  • Superusers bypass Row-Level Security entirely. The drafted Phase 2 RLS bridge (set pgweb.user_id GUC, write USING (author_id = current_setting('pgweb.user_id')::bigint) policies) will silently not enforce anything unless the execution role changes first. Session 6's open question B1 leans "GUC over SET ROLE" — that lean is incompatible with the current connection model. The fix (connect as, or SET LOCAL ROLE to, a dedicated non-superuser pgweb_app role with explicit grants) needs to land before auth ships, and it will surface grant-management work the roadmap hasn't budgeted.

Evidence: crates/pg_web_ext/src/worker.rs:56; PG docs on BackgroundWorkerInitializeConnection NULL-user semantics; docs/internal/sessions/session_6.md Track B.

▶criticalHandlers can't set status codes, headers, cookies, redirects, or content types — and Phase 2 depends on all of these

The response contract is: body string, status 200 (or 404/500 chosen by the framework), Content-Type: text/html always. There is no way for an app to:

  • Issue a redirect (no 302/303 + Location — Post/Redirect/Get is impossible; HTMX swaps paper over this but plain-form fallbacks break),
  • Set or read cookies (so even hand-rolled auth is impossible in userland today),
  • Return application/json — despite docs/APP-LAYOUT.md advertising "JSON APIs" as a raw-text-route use case (prompt 005 documents this docs-vs-reality gap candidly),
  • Control caching for dynamic pages, send 201/204/4xx semantics, etc.

The hidden cost: the Phase 2 spec's login flow says the POST handler "sets cookie via Set-Cookie header in the response" — a mechanism that does not exist. A response-contract redesign (envelope return type, e.g. {status, headers, body} or a pgweb.respond(...) helper) is an unacknowledged prerequisite on the auth critical path, and it touches the router, the HTTP layer, push validation, check, and the docs simultaneously. Better to do it once, deliberately, before Phase 2 — not as a patch during it.

Evidence: crates/pg_web_ext/src/http.rs:120-135 (hardcoded content-type), src/router.rs:46-60 (ServeOutcome has no headers), prompts/005_json_api_and_mcp_support.md, docs/internal/sessions/session_6.md Track A.

▶highPer-request overhead is unmeasured and structurally heavy: full route scan + fresh template parse + 2–3 transactions per request

Each page request: opens a transaction, loads and sorts every route for the method, fetches the template text, runs the handler, parses the template from scratch with Tera::one_off (no parsed-template cache), then opens a second transaction just to read the env setting for livereload injection (and another for assets/errors). All of these are documented decisions with re-evaluation triggers, and at 20 routes none of it matters — but the combination means the 1k req/s success criterion is doing a lot of unverified work. A routes/templates snapshot cache invalidated by LISTEN/NOTIFY (infrastructure that already exists for livereload) is the obvious, cheap v0.3 win.

Evidence: router.rs:382-465 (per-request fetch+sort), templating.rs:19 (one_off), http.rs:127,165,203 (extra env transactions); VISION.md "1,000 req/s — to be benchmarked."

▶highNo graceful-shutdown path for the HTTP worker

The worker registers SIGHUP/SIGTERM wake flags, then parks forever in rt.block_on(axum::serve(...)) — nothing ever checks the signal flags or wires them into Axum's graceful-shutdown future. On pg_ctl stop the postmaster's SIGTERM is likely ignored until escalation, meaning slow shutdowns/restarts of the whole database because of the web worker. Low effort to fix (a with_graceful_shutdown future polling BackgroundWorker::sigterm_received()); meaningful operational papercut until then.

Evidence: crates/pg_web_ext/src/worker.rs:48-112.

Security posture

▶highThe trust model is "anyone who can reach :8080 runs your SQL as superuser, with no rate limit, no CSRF, and no security headers"

Today that's a coherent Phase-1 position for personal apps behind Caddy — there are no sessions to steal and the demo is anonymous-by-design. But several layers are absent at once: no CSRF protection (a cross-site form can POST to any pg-web app; the todo demo is writable by any page you visit), no rate limiting or login throttling (planned only as a documented recipe), no CSP/X-Frame-Options/etc. defaults, and a 2 MiB body cap as essentially the only abuse control. Combined with single-threaded serving, a trivial slow-POST or expensive-query loop is a one-curl denial of service. Phase 2 plans CSRF properly; the rest needs an explicit hardening pass and a threat-model doc — adopters putting an HTTP listener inside their database will ask for exactly that document first.

Evidence: http.rs (no security headers, MAX_BODY_BYTES only), session_6.md CSRF section (planned), DEPLOYMENT.md security checklist (host-level only).

▶mediumSecrets sit in a plain table readable by every handler and every DB session

pg-web env set STRIPE_KEY=… writes plaintext into pgweb.settings. Acknowledged as a trade-off in ARCHITECTURE.md, but the consequences compound with superuser execution: any handler, any SQL injection, any pg_dump, and any future pg-web export can exfiltrate all secrets. Even without encryption-at-rest, separating secrets into a table with column-level grants for a dedicated role (post-role-split), excluding them from dumps/exports by default, and masking env list output would close most of the practical gap.

Evidence: schema.rs:72-77, env.rs, ARCHITECTURE.md "Secrets management."

▶lowHand-rolled SQL quoting on the request path (a documented workaround, but a standing footgun)

Route/template/asset lookups interpolate values via a local quote_literal() (doubling single quotes) instead of SPI parameter binding — the team hit a rustc 1.95 ICE with pgrx's DatumWithOid arrays and documented the workaround (pitfall catalog). The escaping is correct under default standard_conforming_strings=on, and handler names are additionally gated by a strict identifier allowlist before being spliced into the wrapper call. Risk today is theoretical; the cleanup (return to bound parameters when toolchain allows) should be tracked so the workaround doesn't fossilize into five more call sites.

Evidence: router.rs:322-357 (quote_literal / is_safe_ident), schema.rs:131-163 (_framework_call_handler uses EXECUTE … USING for the JSON — good), OVERVIEW.md gotcha #7.

Capability gaps that bite users early

▶highGET and POST only — no PUT/PATCH/DELETE, no file uploads, no Range/compression

The layout scanner reserves put/patch/delete/head/options stems for "Phase 2+", so HTMX's own hx-delete/hx-put attributes are unusable; the demo routes deletion through POST /todos/delete. Multipart bodies aren't parsed (forms only, 2 MiB cap) so file upload — table stakes for CRUD apps — has no story. Assets are served whole from memory with no gzip/brotli and no Range support (video/PDF seeking breaks). Individually small; together they define the line between "demo apps" and "real apps", and most are wiring rather than architecture.

Evidence: paths.rs:141-178 (reserved stems), http.rs:29-32,98-102 (form-only parsing, body cap), http.rs:158-194 (no Range/encoding).

▶medium`pg-web check` validates Postgres SQL with a non-Postgres parser — a structural source of false positives

The offline validator uses the generic sqlparser crate. Real-world Postgres SQL keeps escaping its grammar: two field bugs already shipped and were fixed (dollar-quoted COMMENT ON bodies; gin_trgm_ops opclasses), each patched by carving out tolerant bypasses — CREATE EXTENSION and CREATE INDEX statements now skip parsing entirely, which also means real typos in those statements pass. The team knows: prompt 004 proposes validating against an ephemeral live dev database instead, and the doc comments openly call the checks "intentionally approximate". The upgrade path (libpg_query bindings, or the live-DB preflight) should come before users learn to ignore red checks.

Evidence: check.rs:1-60 (scope disclaimers), commits 10d415d, 30c1235, 3c7fe3b; prompts/004_live_dev_db_for_check_validation.md.

▶mediumOne worker serves one database; multi-app or multi-tenant hosting means one container per app

The BGW connects to a single database chosen by PGWEB_DATABASE/POSTGRES_DB at boot, binds one hardcoded port, and serves that app. Hosting two apps on one Postgres = two clusters/containers. Fine for the single-VPS target persona; a real constraint for the "platform" ambitions (and for anyone hoping to use spare capacity of an existing cluster). Worth an explicit roadmap position: per-database workers with per-database ports/hostnames, or a stated non-goal.

Evidence: worker.rs:20-37.

Operational & consistency issues

▶mediumNo extension upgrade path between versions

There are no pg_web_ext--0.1.0--0.2.0.sql upgrade scripts; installs are bootstrap-only. Today schema churn is invisible (v0.2 changed a CHECK constraint via fresh installs), but the moment a real user has data in a v0.2 container and Phase 2 adds pgweb.sessions, upgrades become "dump, recreate, restore" unless ALTER EXTENSION paths ship. Also note the runtime coupling: upgrading the web server now means replacing the Postgres image and restarting the database — worth documenting as an accepted cost of the architecture.

Evidence: extension SQL emitted only as bootstrap (schema.rs extension_sql!), single versioned install file in the image.

▶mediumNo health endpoint, no metrics, no persisted request log — and the Docker healthcheck runs your homepage

/_pgweb/health and Prometheus metrics are Phase 3/4 items; meanwhile the image's HEALTHCHECK curls GET /, so a bug in the user's index handler marks the whole container unhealthy (restart loops under orchestrators). Request logging exists only as stdout tracing; prompt 006 (access logging) is open. For a framework whose pitch includes "ops visibility" (the deployments ledger is great), the runtime side is still blind.

Evidence: Dockerfile HEALTHCHECK, ROADMAP Phase 3/4, prompts/006.

▶lowDoc/code drift: the configurable port that isn't, and a self-violated invariant

ARCHITECTURE.md says the port is "configurable via the pgweb.port GUC"; the code hardcodes 8080 (worker.rs:21, "will become a GUC later") and pgweb.toml [server].port is read by nothing at runtime. Separately, CLAUDE.md invariant #5 forbids the extension opening TCP connections back to Postgres — which is exactly what the livereload LISTEN task does (tokio-postgres to loopback), with good reasons, dev-only today… but Phase 2 realtime makes that connection always-on in production. The invariant should be amended rather than silently excepted. Minor, but invariant documents only work if they're kept true.

Evidence: worker.rs:21,127-143, listen_router.rs:150, CLAUDE.md §invariants, ARCHITECTURE.md §background worker.

▶lowSmall correctness nits
  • lookup_fallback issues two separate queries for one row's two columns (router.rs:470-486).
  • HEAD / returns the 404 fallback (only GET/POST routes exist; Axum's fallback receives HEAD verbatim) — surprising for probes.
  • Tokio is compiled with rt-multi-thread feature while only the current-thread runtime is used — harmless dependency residue.
  • The ROADMAP admits live-reload SSE lacks automated bfcache/navigation-leak tests after fixing exactly that bug class — the regression test is a string-assertion on the JS, not a behavioral test.
  • OVERVIEW.md's code map lists ROADMAP/ARCHITECTURE twice and omits newer modules (livereload, errors, settings) — cosmetic drift.

Project-level risks

▶highAdoption headwinds: bus factor of one, a skeptical DBA culture, and a narrow eligible audience

The eligible user must: own their Postgres host (no RDS/Cloud SQL/Supabase — custom extensions are banned there), accept HTMX-only UI, write business logic in SQL/PL/pgSQL, and be comfortable running an HTTP listener inside the same process tree as their data. Each filter is defensible; multiplied, they define a niche-of-a-niche. Meanwhile the strongest cultural antibody — "nothing extra runs in my database" — comes precisely from the operations people who'd have to approve it; a bug in the HTTP path lives in the database's process tree (own process, but shared fate on crashes via the postmaster's 5s restart policy, shared CPU/RAM). Add a single anonymous-ish maintainer, zero external contributors, and an agent-driven process with no second reviewer, and the project's survival currently depends on one person's continued attention. None of this is a code flaw; all of it shapes what the project can credibly promise. The mitigations are the classic ones: ruthless honesty docs (already strong), a benchmark + threat model, and making the first 10 external users heroically successful.

6Competitive landscape — where it sits

ProjectThesisHow pg-web differs
SQLPageRust binary beside the DB; .sql files render to server-side pages via prebuilt components.Closest DX cousin. pg-web moves the server inside Postgres (SPI, zero hops), uses free-form Tera+HTMX instead of a component catalog, and stores the app in the database. SQLPage is more mature, multi-DB, and lower-risk; pg-web is more radical and more integrated (migrations, env, check, ledger, live-reload).

Omnigres"Postgres as application platform" — a suite of extensions including an in-database HTTP server.The most direct thesis rival. Omnigres is broader and more infrastructural; pg-web is opinionated and product-shaped: one app model (pages/ + HTMX), one CLI, one image, a tutorial that works. Timing tailwinds: the HTMX/hypermedia revival, the "just use Postgres" movement (queues, search, vectors in PG), and growing distrust of managed-cloud lock-in all blow in pg-web's direction. The headwind is that each of those movements has cheaper on-ramps than installing a C-ABI extension into your database.

7Possibilities — what this could open up

1 · The killer artifact: your entire app is one pg_dump

Because routes, templates, assets, handlers, settings, schema, and data all live in one database, a single dump file is the application — code and state, restorable into a fresh container as a running system. The roadmap's Phase 4 trio (pg-web backup, code-only export, and a pgweb.sources schema mirroring the working tree and even git history into tables) would make pg-web the only web framework where "hand someone a file" equals "hand someone the whole product." This is the most marketable, least-contested idea in the repo: nothing mainstream offers clone-prod-to-laptop as one command. It deserves to be pulled forward from Phase 4.

2 · An agent-native backend (the prompt-005 vision)

The queued JSON API + MCP exploration reframes pg-web for the agent era: the same SQL handlers serving humans HTML could expose typed MCP tools and sparse JSON to LLM agents — making a pg-web app a transactional "memory layer" for agent swarms, with RLS as the permission system and LISTEN/NOTIFY as the event bus. The project's own development process (CLAUDE.md invariants, prompt work orders) doubles as evidence the author understands agent ergonomics. Paired with the parked "docs MCP" idea (agents writing pg-web apps get authoritative, current framework docs as tools), this is a coherent and early position almost no web framework currently occupies.

3 · Database-native realtime with zero extra infrastructure

The channel-aware ListenRouter (one LISTEN connection fanning out to N SSE tabs in memory) already works for live-reload. Phase 2's planned <div hx-ext="sse" sse-connect="/_pgweb/subscribe/orders.user.42"> + pgweb.notify_app() would give CRUD apps push updates with no Redis, no websocket service, no message broker — the database's own pub/sub doing the work. For the target persona (small self-hosted apps), that's a feature normally costing two more containers.

4 · Concrete niches it could own near-term

  • Internal tools & admin panels — SQL-literate teams shipping CRUD against existing Postgres data, one container, behind the VPN.
  • Single-tenant deployments — one customer = one container = one dump; a compliance-friendly isolation story that multi-tenant SaaS frameworks fight to fake.
  • pgvector-era demos and small AI products — embeddings, search, business logic, and the UI in literally one process tree.
  • Teaching — the cleanest "SQL is enough" pedagogy vehicle since PostgREST; the tutorial already reads like a course module.
  • Spin-offs — the planned native-Rust pg_catalog schema-diff engine (Phase 2.5) would be valuable to the whole ecosystem as a standalone crate, framework aside.

10Bottom line

What it is promising: a credible bet that for a meaningful class of applications — small, self-hosted, SQL-centric, hypermedia-driven — the database can absorb the entire web stack, and that the payoff (one artifact, one backup, zero hops, an app you can pg_dump) outweighs the heresy. Phase 1 proves the mechanics; the live dogfooded docs site proves the sincerity; the roadmap proves the author can see most of the gaps without outside help.

What stands between it and that promise, in priority order:

  1. Response contract v2 (headers/status/cookies/content-type) — the hidden prerequisite under Phase 2 and the JSON/MCP future. Highest leverage per line changed.
  2. Privilege floor — non-superuser execution role + per-request timeout + threat model. Without this, the security conversation with any serious adopter ends early, and the RLS bridge won't actually enforce.
  3. Concurrency honesty — benchmark, publish, then either ship the multi-worker design or scope the framework explicitly to low-concurrency apps. Both are respectable; ambiguity isn't.
  4. Auth (Phase 2 as specced) — the gate between "demo framework" and "app framework."
  5. Pull the backup/export story forward — it's the feature nobody else can match and it's currently parked in Phase 4.

Recommended one-line positioning

"The web framework you can pg_dump" — lead with the artifact story and the dogfooded site; let zero-hop latency be the supporting act rather than the headline, since the single-threaded runtime can't yet cash the performance check the vision writes.

Use today for: internal tools, single-tenant deployments, HTMX prototypes, agent-backend experiments, and as a reference for disciplined agent-driven development. Hold off for: anything multi-user (until Phase 2 + role hardening), anything latency-SLO'd or traffic-spiky (until concurrency lands), anything where a DBA must sign off (until the threat model exists).
