## Summary

pg-web must ship **working health and readiness endpoints out of the box** so that `pg-web init`, Docker HEALTHCHECK, load balancers, and orchestrators have something reliable from minute one.

Two distinct surfaces:

1. **Protected platform probes** (always available, never overridable by user routes):
   - `GET /_pgweb/health` — liveness (BGW alive + cheap SPI probe).
   - `GET /_pgweb/readiness` — readiness (liveness + framework schema present + basic version sanity once 018.2 lands).
   - Hard-mounted in Axum (exactly like `/_pgweb/livereload*`) so they win before the user fallback/router lookup. These are the ones the Dockerfile HEALTHCHECK and serious infrastructure should use.

2. **App-level, user-overridable defaults** (what "just works" for normal `/health`):
   - `GET /health`
   - `GET /readiness`
   - If the user has **not** defined a route for the path, the framework serves a sensible default (simple JSON or text body, heavily commented in the implementation).
   - If the user creates their own handler (e.g. `pages/health/index.sql` or `pages/health/index.html` + `.sql` per APP-LAYOUT.md) and pushes it, their route **overrides** the default completely.
   - The public names are the conventional ones (`/health`, `/readiness`) so they are discoverable and work with standard tooling.

Additional controls:
- Settings (via `pgweb.toml` → `pgweb.settings` and `pgweb.setting()`) to **disable the default app-level health/readiness** (`health_enabled`, `readiness_enabled` or equivalent). When disabled, the framework does not serve the public `/health` (falls through to normal 404 / `_404` handling); the protected `/_pgweb/*` probes remain unless explicitly documented otherwise.
- These flags (and the existence of the endpoints) are **documented and present in the output of `pg-web init`** (both minimal and `--template todo`), including comments in the generated `pgweb.toml` and README.

The endpoints are deliberately **lightweight** (a `SELECT 1` or equivalent + settings read, or a pure in-memory check for the protected path). We accept that they run on the single SPI-attached thread (no multi-port or multi-thread work for this feature). The protected probes are cheap enough that HOLB is not a first-order concern for the common "user handler is slow/broken, platform is fine" case.

The original footgun (Docker HEALTHCHECK curling the user's `/`) is eliminated by pointing it at the protected `/_pgweb/health`.

## Current state (verified)

- Dockerfile HEALTHCHECK: `pg_isready && curl -sf http://127.0.0.1:8080/ > /dev/null` (hits seeded or user `GET /`).
- No `/health` or `/_pgweb/health` anywhere in code or seeded schema.
- Framework-reserved `/_pgweb/*` pattern exists and works:
  - `http.rs:86-94`: `livereload_routes` and `static_routes` are built as `Router`s and `.merge()`d **before** `.fallback(handle)`.
  - Comment at http.rs:76-78 explicitly allows a user to still define their own `GET /_pgweb/foo` (it won't collide with the framework ones because of mount order).
  - `livereload.rs` shows the pattern for a framework route that does a `BackgroundWorker::transaction(...)` for settings (and env-gating).
- User routes come exclusively from `pgweb.routes` table lookup inside `router::serve` (normal path).
- New apps are created via `pg-web init` (minimal scaffold in `init.rs` + `templates.rs`) or `--template todo`. `pgweb.toml` is the source of truth for server settings; it is pushed into `pgweb.settings`. `.env.example` is already referenced in the generated `.gitignore`.
- `pgweb.settings` + `pgweb.setting(key)` (seeded for `env` and `request_timeout`) is the established mechanism for runtime flags that handlers and framework code can read.
- Single-threaded current-thread Tokio runtime in the BGW (`worker.rs`); every request (including future health ones that go through SPI) serializes on that thread.

## Proposed work

1. **Add the two protected platform routes (hard Axum mounts).**
   - In `http::app()`, add (alongside livereload):
     ```rust
     let health_routes = Router::new()
         .route("/_pgweb/health", get(health::serve_health))
         .route("/_pgweb/readiness", get(health::serve_readiness));
     ```
   - These handlers are tiny, synchronous-friendly wrappers that do the cheapest possible work (one `BackgroundWorker::transaction` doing `SELECT 1`, or for readiness a quick catalog check + (post-018.2) extension version sanity).
   - Return 200 + small body (`{"status":"ok"}` or plain text) on success; 503 on failure.
   - Keep the module flat (a small `health.rs` or a section inside an existing file — follow "no premature abstraction" until it earns its keep).
   - These routes are **unconditionally available** (not env-gated like livereload). They answer "is the pg-web platform serving HTTP and able to talk to its SPI backend?"

2. **Ship default overridable app-level health/readiness via the normal route mechanism.**
   - Seed (in the bootstrap `extension_sql!` block in `schema.rs`, or via a small dedicated seed function) default rows for:
     - `GET /health` → a framework-owned handler (e.g. `pgweb._default_health_handler` or similar) + optional minimal template or raw response.
     - `GET /readiness` → same.
   - The default handlers must be **heavily commented** explaining:
     - This is the framework default.
     - App authors override by creating their own `pages/health/index.sql` (or `pages/health/index.html` + `.sql`) per the directory-as-route rules.
     - When they push, their route wins because `pgweb.routes` lookup takes precedence (or the router explicitly prefers a user row).
     - The body is intentionally simple today (text or basic JSON); we can make responses richer (status envelope, custom content-type, etc.) later using the v2 response helpers (`pgweb.respond` / `pgweb.json`).
   - If the user never touches `/health`, the seeded default just works after `CREATE EXTENSION` or `pg-web push` (even on a minimal init).
   - The default must be **replaceable**: a user-defined route for the exact path_pattern + method must take over completely (consistent with how the seeded `/` + hello is replaced today).

3. **Disable controls for the app-level defaults.**
   - Add two new keys to the settings surface (synced from `pgweb.toml [server]` like `env` and `request_timeout`):
     - `health_enabled` (default true)
     - `readiness_enabled` (default true)
   - When false, the framework **does not serve the public default** for that path (normal router miss → 404 or `_404` handling applies). User-provided routes for `/health` still work (they are just user routes).
   - The protected `/_pgweb/health` and `/_pgweb/readiness` are **not** disabled by these flags (they are infrastructure probes).
   - Expose the keys with comments in the `PGWEB_TOML` template (both minimal and todo paths).
   - Document the keys in the generated READMEs and in `docs/APP-DEVELOPER-GUIDE.md` / deployment docs.
   - Handlers (and the default health functions) can read the flags via `pgweb.setting('health_enabled')`.

4. **Point the Dockerfile HEALTHCHECK at the protected probe.**
   - Change line ~119 from `curl -sf http://127.0.0.1:8080/ > /dev/null` to `curl -sf http://127.0.0.1:8080/_pgweb/health > /dev/null`.
   - Update the comment above it.
   - This completely removes the footgun: a broken or slow user `GET /` (or a user-overridden `/health` that 500s) no longer makes the container unhealthy.

     - Optionally seed a minimal `.env.example` (POSTGRES_* + note about health flags living in pgweb.toml) if we decide to start committing one; at minimum update `.gitignore` handling and READMEs if needed.

## Constraints & invariants (cite CLAUDE.md)

- **Single-thread acceptance:** These endpoints (especially the protected ones) are intentionally trivial. We do **not** introduce a second listener/port or multi-threaded runtime for health in this work order. The user has explicitly accepted the lightweight-on-the-request-thread trade-off.
- **`_pgweb/*` is the protected namespace.** The platform liveness/readiness routes **must** be mounted via `.merge()` above `.fallback` exactly like livereload. User code may still define routes under `/_pgweb/`; framework ones win by mount order.
- **Extension ↔ CLI decoupled.** All serving logic (defaults, protected handlers, disable checks) lives in the extension. The CLI only scaffolds files and does the normal `pgweb.toml` → `pgweb.settings` push. No new shared types needed.
- **One request = one SPI tx.** Any SPI done by the health handlers (settings read, `SELECT 1`, catalog check) happens inside the normal per-request transaction boundary.
- **No `unwrap`/`expect` on request path.**
- **Companion app + comments:** The override pattern must be exercised (or clearly documented) in `examples/todo/` with high-quality explanatory comments that teach a new developer how to provide their own health logic.
- **PG 15/16/17:** Any new seeded SQL (default handlers, settings keys) must be valid on all three.

## Open questions (for the implementing session)

1. **Seeding vs. router fallback for the overridable defaults.** Seed rows in the bootstrap SQL (like the hello route) so `pg-web push` can simply overwrite them the normal way? Or have the router do "if lookup misses for exactly /health and the setting is enabled, synthesize the default response"? Seeding is more consistent with existing behavior; a special-case in the router is "magic" but avoids polluting the routes table on every fresh install.
2. **Exact setting key names and defaults.** `health_enabled` / `readiness_enabled` (booleans, default true)? Or a single `default_health_endpoints = "both"|"health"|"readiness"|"none"`? Lean: two booleans for simplicity.
3. **Body format for the defaults.** Plain text `ok` for now? Or small JSON `{"status":"ok","pgweb":"..."}`? User said "heavily commented that we can make it be json only when we want later" — so start simple and call the evolution path out in comments.
4. **Should disabling also affect the protected probes?** Probably not — the container still needs to know if the BGW is alive. Document the choice.
5. **Readiness version check depth.** How much to do in 018.1 vs. waiting for 018.2? Lean: cheap catalog existence now; richer "extension version matches expected" after the upgrade path exists.
6. **.env.example in scaffolds.** Do we start emitting one from `pg-web init` (with POSTGRES_PASSWORD example + comments pointing at pgweb.toml for health flags)? The gitignore already anticipates it.
