## Summary

Today the framework schema is installed **only** as a `bootstrap` `extension_sql!` block. pgrx emits a single `pg_web_ext--<ver>.sql` install file. The Dockerfile copies only install artifacts. No `pg_web_ext--A.B--C.D.sql` upgrade scripts exist.

`docs/DEPLOYMENT.md` already *documents* the command `ALTER EXTENSION pg_web_ext UPDATE;` and claims Postgres will run the migration script — but the script is not shipped. Any schema change after a user has real data (e.g. the `pgweb.assets` CHECK cap 2→20 MiB, future Phase-2 tables) forces a dump/recreate/restore today.

This work order establishes the real `--from--to` upgrade convention, the policy for safe changes, a test tier that exercises in-place upgrades (on PG 15/16/17), and honest documentation of the inherent cost of the model (framework upgrades require a Postgres restart because the web server lives inside the BGW).

This is the time-sensitive half of the original 018. Do it before (or alongside) 018.1 if user data + new framework schema changes are imminent.

## Current state (verified)

- `crates/pg_web_ext/src/schema.rs:13-470` is one giant `extension_sql!(..., name = "framework_tables", bootstrap)`.
- `bootstrap` means "runs first on CREATE EXTENSION" — it is an *install* artifact.
- `pg_web_ext.control` has `default_version = '@CARGO_VERSION@'`.
- `Dockerfile:97-99` copies the `.so`, the `.control`, and `pg_web_ext--*.sql` (only the install file).
- `docs/DEPLOYMENT.md:135-144` describes the intended flow (`docker compose pull` → `up -d` → `ALTER EXTENSION ... UPDATE`) and asserts the migration script will be executed. The file does not exist.
- Past schema evolution (assets cap, new settings keys, new helper functions, response-contract v2 helpers, etc.) has only ever been exercised on fresh `CREATE EXTENSION` or clean containers.
- Because the HTTP worker is a Postgres background worker (`worker.rs`), replacing the `.so` requires replacing the container image and restarting the Postgres process. There is no independent rolling web deploy.

## Architectural coupling that must be documented

Framework upgrades are **not** zero-downtime in the same way app pushes are.

- `pg-web push` (routes/templates/handlers/assets) **is** zero-downtime: new traffic picks up the new rows on the next SPI lookup.
- Framework upgrades (new `.so` + schema changes) are **not**: they require an image swap + `docker compose up -d` (or equivalent) + a Postgres restart. The docs currently blur this distinction.

This is an inherent property of "the database is the web server." Users coming from stateless web tiers will be surprised if we do not call it out plainly.

## Proposed work

1. **Establish the `--from--to.sql` upgrade-script convention.**
   - Research (see below) exactly how pgrx 0.18 emits (or can be made to emit) upgrade scripts alongside the bootstrap install SQL.
   - Create the build + packaging convention so that `pg_web_ext--0.2--0.3.sql` (etc.) files are produced and land in `/usr/share/postgresql/${PG_MAJOR}/extension/` inside the runtime image (same place the Dockerfile already copies install artifacts).
   - Confirm that Postgres + `ALTER EXTENSION pg_web_ext UPDATE` correctly resolves and runs a chain of scripts when jumping versions (e.g. 0.2 → 0.4 runs 0.2→0.3 then 0.3→0.4).
   - The `.control` `default_version` already tracks the cargo version; the upgrade mechanism must integrate with it.

2. **Additive vs. destructive schema change policy.**
   - Document (in `CLAUDE.md` and a short section in `docs/DEPLOYMENT.md` or a new `docs/UPGRADING.md`) the rule:
     - **Additive** (new tables, new nullable columns, widening CHECKs or domains, new functions, new settings keys, new GRANTs, etc.) are always safe. Put the exact DDL into the upgrade script verbatim.
     - **Destructive** (DROP, narrowing constraints, type changes that can fail on existing rows, removing columns, etc.) require an explicit migration strategy with data-preservation steps + a breaking-change note in the upgrade script and changelog.
   - Canonical worked example: the `pgweb.assets` CHECK cap change (2 MiB → 20 MiB) was additive (widening) and therefore safe.
   - **Lean decision:** no downgrade scripts (`ALTER EXTENSION ... UPDATE TO <older>`). Downgrades are a deep rabbit hole; document "restore from backup" as the supported rollback path. State this explicitly.

3. **New test tier that exercises real in-place upgrade.**
   - Today's tiers never do an upgrade of a running instance with user data.
   - Add coverage (tier-3 style, run on PG 15, 16, and 17 per invariant #6) that:
     - Boots the *previous* published image (or installs the previous `--<ver>.sql` install file).
     - Runs `pg-web push` (or direct inserts) to create app routes + user data + `pgweb.deployments` rows.
     - Swaps to the new image + new `.so` + the new upgrade script(s).
     - Executes `ALTER EXTENSION pg_web_ext UPDATE;`.
     - Asserts:
       - Framework tables migrated (new columns, new objects present).
       - All prior user data, `pgweb.deployments`, user tables, and routes/templates/assets are intact.
       - The app still serves (basic HTTP smoke + the pushed routes).
   - **Lean first iteration:** a self-upgrade smoke inside a single test run is acceptable — "install version N's SQL, hand-apply a synthetic `--N--N+0.0.1.sql` that adds a column or setting, run `ALTER EXTENSION UPDATE`, assert the new object exists and seeded rows survived." Full previous-image pull testing can be added once two real published versions exist.
   - This test becomes the gate that prevents "we only ever test fresh installs" regressions.

4. **Rewrite the upgrade procedure docs.**
   - Update `docs/DEPLOYMENT.md:135-144` (the "Upgrading the framework itself" section) so that:
     - It is backed by actually-shipped scripts.
     - It states the DB-restart cost plainly and distinguishes it from zero-downtime `pg-web push`.
     - It recommends `pg_dump` before upgrading (the backup recipe already lives later in the same file).
     - It cross-references the new policy (additive vs destructive) and the test tier.
   - Also update `CLAUDE.md` (add the policy under coding practices or architectural invariants) and `docs/ROADMAP.md` / `docs/OVERVIEW.md` as needed.
   - Add an explicit note in the architecture / deployment story: "App data pushes are zero-downtime. Framework (extension) upgrades require a Postgres restart because the HTTP worker lives inside the database process."

5. **(Stretch / dogfood)** Make one of the new settings or tables added in this era (or the readiness version check from 018.1) the first real customer of an upgrade script.

## Constraints & invariants

- **#3 Extension ↔ CLI strictly decoupled.** Upgrade scripts and the extension build are purely extension + image concerns. The CLI never emits or reads upgrade SQL.
- **#4 One HTTP request = one SPI transaction.** (Not directly relevant to upgrade scripts, but any new framework objects must respect the per-request tx rule once deployed.)
- **#6 PG 15/16/17 only.** Upgrade scripts and any new schema must be valid (and the test tier must pass) on all three supported majors. No pg18-only facilities.
- **Additive by default in Phase 1/2.** Keep changes additive where possible so the upgrade story stays simple.
- **No premature abstraction.** The first upgrade script(s) can be small and hand-maintained. Do not invent a full schema-diff engine here (that is explicitly Phase 2.5 per ROADMAP).
- **Honesty in docs.** We must not repeat the current situation where DEPLOYMENT.md promises a mechanism that doesn't exist.
