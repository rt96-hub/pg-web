## TL;DR of the ask

1. **Serve on K processes, not one.** Register K background workers behind `SO_REUSEPORT` so the kernel load-balances connections; each worker keeps its own SPI backend + single-thread runtime. A slow request stalls 1 of K, not the whole site. This is the throughput + head-of-line-blocking (HOLB) fix.
2. **Keep a LISTEN/NOTIFY connection per worker (hard requirement).** Every worker must keep its own `LISTEN` connection so a Postgres `NOTIFY` reaches *all* workers, and each worker fans it out to the clients the kernel routed to it. This is what makes livereload / realtime updates reach **all users and sessions regardless of which worker served them** — and it must hold in production, not just dev. (Production livereload itself still has a way to go — see §5 — but the per-worker LISTEN is the non-negotiable foundation and must not regress when we go multi-worker.)
3. **Make the worker count a config in `pgweb.toml`** (the user-set override). **Phase 1 (this prompt):** an explicit `workers = N` knob. **Phase 2 (explore after):** autoscale to available resources by default, with the `pgweb.toml` value as the override. Start with the config; layer autoscale on top later.

The benchmark numbers (below) say this is now urgent, not theoretical.

## Why now — the honest HOLB data (from 030)

015 Step 1 built the harness; 030 fixed a worker-self-termination bug (`729eb93`) that had been **masking** the real behavior (the bench read "72%-then-0%" because the worker died 8s after boot — not "single-worker reality"). With that fixed, the single-worker serialization is now measured and dramatic:

- **HOLB headline** (`docs/BENCHMARKS.md`, `bench` `d-fast-under-slow` vs `b-todos100-c16-pure`): the fast `/bench/todos` c=16 path runs at **p99 ≈ 3.7–4.2 ms** alone, but **p99 ≈ 218–222 ms** when a low-rate (`-q 3`) `pg_sleep(0.2)` injector runs concurrently — **~60× tail blow-up** on requests whose own queries are microseconds. This happens **even on the unconstrained full-host tier**: extra cores buy nothing because one thread serves everything.
- **Pure serialization** (`030` `BENCH_SELFTEST` probe): 16 concurrent `pg_sleep(0.2)` requests against one worker → **~5–6 req/s, p99 ≈ 3.3 s** — i.e. the 16 clients are serviced strictly one-at-a-time on the single SPI thread.

The bench gate (030) makes the slow-handler experiment a permanent, automated before/after: once K>1 lands, `d-fast-under-slow`'s p99 should stay near its no-interference baseline (one worker takes the sleeper; the rest serve fast). That flip is the acceptance proof (§9).

## The hard constraint (physics, not preference)

Only the OS thread attached via `connect_worker_to_spi` may issue SPI calls. A multi-thread Tokio runtime would migrate a task onto a thread with no SPI backend → panic on the first `Spi::*`. So:

- `worker.rs:145-147` builds `tokio::runtime::Builder::new_current_thread()` **by necessity**, and per-request SQL runs **synchronously, inline** on that one thread: `http::handle` (`http.rs:108`) → `router::serve` (`http.rs:160`) → `BackgroundWorker::transaction(serve_in_tx)` (`router.rs:76-79`), nothing `await`ed around it.
- **Therefore parallelism comes from more *processes*, each with its own SPI backend + own single-thread runtime — never more threads in one process.** This is why "just use a multi-thread runtime" and "`spawn_blocking` the SQL" are both off the table. (CLAUDE.md invariant #7; `worker.rs:145-147`.)

## Design

### Option A — K workers behind `SO_REUSEPORT` (the throughput + isolation mechanism)

Register **K** workers; each is its own postmaster-owned process, calls `connect_spi_as_serving_role` for its **own** SPI backend, builds its **own** `new_current_thread()` runtime, and binds `0.0.0.0:8080` with **`SO_REUSEPORT`**. The kernel load-balances accepted connections across the K listeners. Nothing is shared but the port + the database (all serving state already lives in PG). Invariant #4 ("one request = one SPI transaction") holds **per worker**.

Change surface (with current refs):
- **Registration loop** — `lib.rs:42-48`: loop `i in 0..K`, name each `pg_web_worker_{i}`, pass the index via `.set_argument(Some(i.into_datum()))`, keep `.enable_spi_access()` + `.set_restart_time(5s)`. Read K from the GUC (below) before the loop.
- **`SO_REUSEPORT` bind** — `worker.rs:159-168`: build the socket with `socket2` (`Socket::new` → `set_reuse_port(true)` + `set_reuse_address(true)` → `bind` → `listen`), convert to `tokio::net::TcpListener::from_std`. Confirm `socket2` is acceptable (small, ubiquitous) and works on the Linux target (the VPS/Docker kernel). The worker index (`worker.rs:99` `_arg`) is for logging/diagnostics — the port is shared, not per-worker.
- **Everything else per-worker is already correct**: each worker already runs its own runtime, SPI connect, ListenRouter, LISTEN loop, and graceful-shutdown select (`worker.rs`). Going multi-worker mostly means *doing this K times* — which is exactly why the per-worker LISTEN (next section) gives cross-worker realtime "for free."

### The per-worker LISTEN realtime fan-out (HARD REQUIREMENT — must not regress)

Postgres delivers a `NOTIFY` to **every** backend that is `LISTEN`ing on the channel. Because each of the K workers runs its own `run_listen_loop` (`worker.rs:206`), a single `NOTIFY pgweb_livereload …` (from `pg-web dev` post-push), `NOTIFY pgweb_reload …` (cache invalidation on push, already prod), or a future app channel wakes **all K** workers; each worker's local `ListenRouter` then broadcasts to the SSE/long-poll clients that `SO_REUSEPORT` routed to *it*. Net: **every connected client sees the update regardless of its worker** — the property the owner requires for livereload and realtime.

Requirements:
- Every worker keeps its own LISTEN connection (the `+1 backend slot per worker` cost is real and permanent — budget it, §6). **Do not** introduce a single shared listener or a "K=1 in dev" shortcut — both were rejected in 015 and would break the guarantee.
- A slow SPI handler on a worker must **not** starve that worker's LISTEN loop / SSE keepalives. The LISTEN loop is a separate spawned task and the SSE handlers are direct Axum paths that bypass `router::serve`, but verify this holds under the HOLB load (a blocked worker must still forward NOTIFYs to its own clients) — ties into Option B.
- This holds in **production** too: the LISTEN loop is already always-on in prod (`worker.rs:170-178`), driven by the cache channel. (The livereload *SSE endpoint* `livereload.rs:155-161` is still env-gated to dev, and the dev script injection is dev-only — see §5.)

### Worker-count config (the user-set override) — `pgweb.toml` → GUC → K

The blocker to understand: **worker registration happens in `_PG_init` at postmaster start, before the database is up or the extension is loaded.** So worker count **cannot** ride the request-time `pgweb.settings` table (the mechanism `env` / `request_timeout` use, read via SPI during a request — `settings.rs`, `cache.rs`). It must reach `_PG_init` as a Postgres GUC (postgresql.conf / `ALTER SYSTEM` / `-c`) or an env var.

Phase 1 (this prompt):
- **Define a custom GUC `pgweb.workers`** (int) in `_PG_init` via pgrx's `GucRegistry` (this would be the project's *first* GUC — `lib.rs`). Read it before the registration loop.
- **Surface it in `pgweb.toml`** as the user-facing knob, e.g. under `[server]`:
  ```toml
  [server]
  port = 8080
  env  = "production"
  # Number of HTTP worker processes. Each is one PG backend (SPI) + one
  # LISTEN backend. Changing this requires a Postgres restart (workers are
  # registered at postmaster start). Omit/empty => framework default.
  workers = 4
  ```
- **CLI plumbing**: the `pg-web` CLI already parses `pgweb.toml` (`crates/pg_web_cli/src/env.rs`, `stack.rs`). Unlike `env`/`request_timeout` (synced into `pgweb.settings`), `workers` must be written where the postmaster reads it at boot — e.g. `ALTER SYSTEM SET pgweb.workers = N` (persisted to `postgresql.auto.conf`) on `pg-web up`/deploy, or rendered into the container's `postgresql.conf`/`-c` arg. Decide the exact mechanism and document it.
- **Restart semantics (must be loud & honest):** changing `pgweb.workers` requires a **full Postgres restart**, because registration is once-per-postmaster-start. This is unlike `env`/`request_timeout` (live, table-read, picked up on next request/`pgweb_reload`). Document the difference in `pgweb.toml` comments, `DEPLOYMENT.md`, and CLI output (`pg-web push` should note "workers changed → restart required").

### Worker-count autoscale (Phase 2 — explore after Phase 1 lands)

End state the owner wants: **autoscale by default, `pgweb.toml` overrides.** Deferred but scoped here so Phase 1 doesn't paint us into a corner:
- When `pgweb.workers` is unset, default K = a **cgroup-aware** detection of available CPUs. `std::thread::available_parallelism()` already respects Linux cgroup CPU quotas on modern glibc (so a `--cpus=1` container reports 1) — this is the clean primitive and avoids parsing `/sys/fs/cgroup/cpu.max` by hand. Cap it (e.g. `min(detected, sane_max)`) and floor at 1.
- Explicit `pgweb.workers = N` always wins over autoscale.
- Re-validate the budgeting math (§6) for the autoscaled K on small boxes (a 1-vCPU box → K=1, which still benefits from Option B's overload valve and 014's timeout).
- Phase ordering matters: ship the explicit config first (predictable, testable), then make "unset ⇒ autoscale" the default once the budgeting + detection are proven.

### Option B — per-worker bounded queue / overload valve (compose, don't skip)

Per worker, put a bounded semaphore in front of `router::serve`'s `BackgroundWorker::transaction` so the runtime keeps serving cheap non-SPI fast paths (livereload JS stub, SSE keepalives, 404s) while a **bounded** number of SPI calls queue; when full, return **`503` + `Retry-After`** instead of unbounded pile-up. Be honest: this adds **no parallelism** within a worker (SPI is still one-at-a-time) — it buys **graceful degradation**. This is the `ROADMAP.md` "Internal concurrency management" deliverable. Implement per-worker; it composes with A and with 014.

### Not available — offloading SQL (`spawn_blocking`)

Moving handler SQL off the SPI thread is impossible (SPI thread affinity, above). External-API blocking inside handlers remains a real limitation; the answer stays the Phase-3 async job queue (`pgweb.jobs`), not this prompt.

## Connection budgeting (call out loudly)

Per worker = **1 SPI backend + 1 LISTEN backend = 2 backend slots**, plus 1 `max_worker_processes` slot. So K workers = **2K `max_connections` slots permanently held + K `max_worker_processes` slots** (alongside Postgres's own bg workers). The image ships PG defaults (`max_connections=100`, `max_worker_processes=8` — `docker/init-pgweb.sh`), so:
- K=4 → 8 conn slots + 4 worker slots: fine on defaults.
- Larger K (or autoscale on a big box) needs the image to bump `max_worker_processes` (and possibly `max_connections`), and leaves fewer connections for app `pg-web push`/psql. Document recommended bounds + warn when `pgweb.workers` approaches a fraction of `max_worker_processes`/`max_connections`. Update `DEPLOYMENT.md` + `CLAUDE.md`.

## Constraints & invariants (CLAUDE.md — do not violate)

- **#7 Async only in the BGW** — parallelism via more workers, never a multi-thread runtime / tokio in `#[pg_extern]`.
- **#4 One request = one SPI transaction** — preserved per worker under A; B must not split a request across transactions.
- **#5 Zero network hop on the serving path** — serving stays on SPI. The per-worker LISTEN `tokio-postgres` conn (`worker.rs:299-315`) is the sole out-of-band exception and stays off the request path.
- **SPI thread affinity** (`worker.rs:145-147`) — the hard physics; any SPI-on-migrated-thread design is wrong by construction.
- **#6 PG 15/16/17** — `SO_REUSEPORT` via `socket2` must compile on all three; only PG17 must be *correct* at runtime (the NOLOGIN serving-role bypass is pg17-only — `worker.rs:80-85`).
- **Phase discipline** — multi-worker + bounded queue are Phase-3 "Async & Scale". The benchmark/gate (015 Step1/030) were phase-neutral and already shipped.

## Acceptance criteria

1. **K workers register** behind `SO_REUSEPORT`, each with its own SPI backend + single-thread runtime, controlled by **`pgweb.workers`** (GUC) surfaced via **`pgweb.toml` `[server] workers`**; `SO_REUSEPORT` verified to load-balance on Linux. Default K documented.
2. **Restart semantics documented + surfaced**: changing `workers` requires a Postgres restart; CLI/docs say so clearly (contrast with live `env`/`request_timeout`).
3. **HOLB flip (the headline proof):** re-run the 030 bench — with K>1, `d-fast-under-slow` p99 stays near its no-interference baseline (a single slow handler occupies one worker; the rest serve fast). Capture before/after in `docs/BENCHMARKS.md`. Consider a `BENCH_STRICT` ceiling that asserts the relief.
4. **Cross-worker realtime holds:** with K>1, a `NOTIFY` (livereload or a scripted channel) reaches SSE clients on **every** worker — demonstrate with concurrent EventSource clients spread across workers + a triggering push, including while one worker is busy on a slow handler (its peers still deliver). Per-worker LISTEN slot cost confirmed (2K total) and budgeted.
5. **Per-worker bounded queue (Option B)** returns `503` + `Retry-After` under overload instead of unbounded pile-up; demonstrated in the bench.
6. **Budgeting documented** in `DEPLOYMENT.md` + `CLAUDE.md` (prod slots = 2K), image `max_worker_processes`/`max_connections` adjusted/recommended for the supported K range.
7. **Idempotency + gates intact:** full `RUN_BENCH=1 scripts/test-all.sh` green before & after (per CLAUDE.md startup gate); no test tier weakened; companion behavior exercised.
8. *(Phase 2, separate follow-up)* **Autoscale default** when `pgweb.workers` is unset (cgroup-aware `available_parallelism`, capped/floored), `pgweb.toml` override wins.

## Open questions

1. **Default K** (Phase 1): a small fixed number (e.g. 4) vs 1 until autoscale lands? Lean: a conservative fixed default + explicit config now; autoscale default in Phase 2.
2. **`pgweb.toml workers` → boot config mechanism:** `ALTER SYSTEM SET pgweb.workers` (persists to `postgresql.auto.conf`, survives restart, CLI-driven) vs rendering `postgresql.conf`/`-c`. Lean: `ALTER SYSTEM` on `up`/deploy + a `pg-web push` notice that a restart is required.
3. **macOS dev parity:** `SO_REUSEPORT` is last-bind-wins on macOS (not load-balanced). Keep K workers on mac (NOTIFY delivery to all listeners is still correct, so realtime fan-out is testable), document the accept-distribution difference, validate true load-balancing under Linux CI. Do **not** reintroduce a "force K=1" path.
4. **Production livereload (the "ways to go" part):** the per-worker LISTEN foundation is in place, but the livereload SSE endpoint is dev-gated (`livereload.rs:155-161`) and the v1 carrier is a full-page reload / CSS-bust. What's the minimal step to make livereload (or a first realtime channel) usable in production for all sessions — un-gate the SSE endpoint behind a flag? a dedicated `/_pgweb/subscribe/*`? targeted OOB-swap fragments that don't clobber in-progress forms/focus/scroll? Scope separately; this prompt only guarantees the per-worker LISTEN fan-out it needs.
5. **Poison-pill resilience:** with per-worker `set_restart_time(5s)`, can a reliably-panicking request march through and crash all K workers in turn? Do invariant-#4 rollback + 014's timeout cover it, or add a panic catch / per-worker circuit breaker?
6. **Health probes under `SO_REUSEPORT`:** a probe hits a random worker. Keep `/_pgweb/health` (018.1) a fast non-SPI path (bypassing Option B's queue) so a saturated worker still answers liveness.
