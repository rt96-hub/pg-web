# pg-web — App Developer Guide

For developers *using* pg-web to build apps. You write HTML and SQL. You never write or compile Rust. This guide is your contract.

## Install

```
brew install pg-web          # macOS
cargo install pg-web-cli     # any platform with Rust
```

Plus Docker (for running Postgres+extension locally).

## Create a project

```
pg-web init my-blog
cd my-blog
docker compose up -d          # starts the pg-web/postgres container
pg-web dev                     # file watcher + hot reload
open http://localhost:8080
```

That's it. You're serving HTMX-driven HTML backed by your SQL.

## Project layout

```
my-blog/
├── pages/                  # File-system routed (Next.js-style)
│   ├── index.html
│   ├── index.sql           # JSON handler for index.html
│   └── posts/
│       ├── index.html
│       ├── index.sql
│       ├── [id].html       # Dynamic route
│       └── [id].sql        # Handler receives :id as SQL parameter
├── public/                 # Static assets — synced into Postgres
│   ├── styles.css
│   └── logo.png
├── migrations/             # Auto-generated SQL migration files
├── schema.prisma           # Declarative schema (format TBD — may be DBML or raw SQL)
├── pgweb.toml             # Framework config
└── docker-compose.yml      # Generated by `pg-web init`
```

## Routing

| File | URL |
|---|---|
| `pages/index.html` + `pages/index.sql` | `GET /` |
| `pages/about.html` + `pages/about.sql` | `GET /about` |
| `pages/posts/[id].html` + `pages/posts/[id].sql` | `GET /posts/:id` |
| `pages/posts/index.html` | `GET /posts` |
| `pages/_404.html` *(convention)* | 404 error page |
| `pages/_500.html` *(convention)* | 500 error page (prod mode) |

The `.sql` file is optional — if a route has only `.html`, the template renders with an empty context.

## The SQL contract

Each `.sql` file defines a function. The function name must match the file path (normalized). It returns a JSON object, which becomes the template context.

```sql
-- pages/posts/[id].sql
CREATE OR REPLACE FUNCTION pages__posts__by_id(id int)
RETURNS json AS $$
  SELECT json_build_object(
    'post', (SELECT row_to_json(p) FROM posts p WHERE p.id = $1),
    'comments', (SELECT json_agg(c ORDER BY c.created_at) FROM comments c WHERE c.post_id = $1)
  );
$$ LANGUAGE sql STABLE;
```

Inside `pages/posts/[id].html`:

```html
<article>
  <h1>{{ post.title }}</h1>
  <p>{{ post.body }}</p>

  <h2>Comments</h2>
  {% for c in comments %}
    <div class="comment">{{ c.body }} — <em>{{ c.author }}</em></div>
  {% endfor %}
</article>
```

## Hot reload (`pg-web dev`)

On save of a `.sql` file, the CLI:

1. Parses your SQL.
2. Runs it inside `BEGIN; CREATE OR REPLACE FUNCTION ...; ROLLBACK;` to check syntax without committing.
3. If clean, runs the real `CREATE OR REPLACE FUNCTION` in the live database.
4. Prints `✓ pages/posts/[id].sql` to your terminal.

If syntax is broken, you see the exact Postgres error in the terminal. The broken code never reaches the live routing table.

On save of a `.html` file: UPSERTs into `templates`. No restart needed.

On save of a `.css`/`.png`/etc. in `public/`: UPSERTs into `_pg_web_assets_*`.

## Schema migrations

```
pg-web migrate create add_posts
pg-web migrate apply
```

`migrate create`:

1. Parses your schema source file.
2. Introspects the live database.
3. Writes the diff as `migrations/0042_add_posts.sql`.
4. Opens it in `$EDITOR` for you to review before applying.

`migrate apply`:

1. Lists pending migrations.
2. Runs each in a transaction.
3. Records success in `pgweb.migrations`.

## Secrets

```
pg-web env set STRIPE_API_KEY=sk_test_...
pg-web env set DATABASE_URL=postgres://...
pg-web env list
pg-web env unset STRIPE_API_KEY
```

Read in SQL:

```sql
SELECT current_setting('pgweb.STRIPE_API_KEY');
```

## Forms & HTMX validation

Let Postgres constraints do the validation. Catch exceptions in PL/pgSQL, return targeted HTML:

```sql
-- pages/signup.sql (POST handler)
CREATE OR REPLACE FUNCTION pages__signup__post(email text, password text)
RETURNS text AS $$
BEGIN
  INSERT INTO users(email, password_hash)
  VALUES (email, crypt(password, gen_salt('bf', 12)));
  RETURN '<div hx-swap-oob="true" id="message">Welcome!</div>';
EXCEPTION
  WHEN unique_violation THEN
    RETURN '<div hx-swap-oob="true" id="email-error">Email already taken</div>';
  WHEN check_violation THEN
    RETURN '<div hx-swap-oob="true" id="form-error">Invalid input</div>';
END;
$$ LANGUAGE plpgsql;
```

`hx-swap-oob="true"` lets HTMX update named page regions without a full reload.

## Deploying to production

```
pg-web push --url postgres://deploy:pw@db.example.com:5432/myapp
```

The CLI:

1. Connects to the remote Postgres.
2. Runs any pending migrations.
3. Upserts routes, templates, and assets.
4. Commits atomically — no partial deploy state.

Zero downtime. No container rebuild. No rolling restart. All application state lives in the database.

## Configuration (`pgweb.toml`)

```toml
[server]
port = 8080           # Port the extension binds HTTP on
env = "development"   # "development" | "production" — affects error page style

[database]
url_env = "DATABASE_URL"

[assets]
large_cutoff_bytes = 1048576

[dev]
watch_paths = ["pages", "public"]
```

## Things you do NOT need to do

- Write Rust.
- Compile anything.
- Manage a connection pool.
- Configure a separate app server.
- Set up a reverse proxy for local dev.
- Handle CSRF (auto, Phase 2).
- Wire up an ORM.
- Worry about N+1 queries from an ORM (there is no ORM).

## Things you DO need to do

- Own a VPS for production (managed-DB services don't support custom extensions).
- Write SQL.
- Write HTML.
- Think about transactions (each request is one — `BEGIN` to `COMMIT` automatic).
- Set up Postgres RLS policies in Phase 2+ if you have multi-tenant data.
